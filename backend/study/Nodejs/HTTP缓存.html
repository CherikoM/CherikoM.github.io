HTTP缓存

发表于2024-11-08|更新于2024-04-12|后端学习
|字数总计:1.3k|阅读时长:5分钟|阅读量:

来源：Node.js

HTTP缓存主要有强缓存和协商缓存两大类，它们都通过HTTP响应头控制，能够提升网站性能

强缓存

数据被强缓存之后，下次不需要再向服务器发送请求，而是从浏览器缓存读取

强缓存有两种

  1. memory cache内存缓存：将缓存存储在浏览器内存当中，通常刷新网页时会发现很多内存缓存
  2. disk cache硬盘缓存：将缓存是存储在计算机硬盘中，空间大，但读取效率比内存缓存慢

强缓存方式由浏览器根据某种策略决定，用户设置不了

Expires

响应头的Expires字段可以指定响应的到期时间（资源不再被视为有效的日期和时间），它是一个HTTP1.0的头部字段，但仍被一些客户端和服务器使用

Expires的判断机制：

1. 当客户端请求资源时，获取本地时间戳
 2. 拿本地时间戳与Expires设置的时间做对比
 3. 如果对比成功，走强缓存，失败则对服务器发起请求

Cache-Control

Cache-Control有很多key

  • max-age：浏览器资源缓存的时长（秒）
  • no-cache：不走强缓存，走协商缓存
  • no-store：禁止任何缓存策略
  • public：资源既可以被浏览器缓存，也可以被代理服务器缓存（CDN）
  • private：资源只能被客户端缓存

如果Cache-Control max-age和Expires同时出现，则采用max-age

协商缓存

强缓存优先于协商缓存，如果强缓存未命中（例如max-age过期），或者服务器响应中设置了Cache-Control: no-cache，则客户端会发起协商缓存请求

协商缓存中，客户端会发送带有缓存数据标识的请求头部字段，以向服务器验证资源的有效性，服务器会根据客户端发送的协商缓存字段（If-Modified-Since/If-None-Match）判断资源是否发生变化

如果资源未发生修改，服务器会返回状态码304（Not Modified），通知客户端可以使用缓存的版本；如果资源已经发生变化，服务器将返回最新的资源，状态码为200

Last-Modified

服务器通过Last-Modified响应头告知客户端资源的最后修改时间

客户端在后续请求中通过If-Modified-Since请求头携带该时间，服务器判断资源是否有更新

如果没有更新，则应用缓存，返回304状态码

ETag

服务器通过ETag响应头给资源生成一个唯一标识符

客户端在后续请求中通过If-None-Match请求头携带该标识符

服务器根据标识符判断资源是否有更新，如果没有更新，返回 304 状态码
